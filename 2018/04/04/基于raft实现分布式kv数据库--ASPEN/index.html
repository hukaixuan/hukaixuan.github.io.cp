基于raft实现分布式kv数据库--ASPEN

2018-04-04

Overview

最近在看分布式系统的一致性算法，基于raft实现了一个简单的分布式kv数据库ASPEN，梳理一下实现过程，总结一下raft算法
主要写一下leader的选举和日志的复制

  • 节点间的消息传递没有采用同步的rpc请求，而是基于UDP，发送方发送UDP请求，接收方监听请求，执行相应逻辑后再通过UDP请求回送给发送方。这种方式的好处是请求和响应的分离；UDP不保证送达，更像真实环境并且不用处理各种响应超时；异步，不会堵塞在RPC请求上

Leader election

raft 算法规定了集群中节点的三种状态(state):

  • follower: 响应candidate的选举请求(requestVote)和leader的appendEntries请求，超过一定时间收不到leader的心跳便成为candidate
  • candidate: 一旦成为candidate便发起requestVote请求，收到多数票则成为leader
  • leader: 接受client命令请求，通过heartbeat下发给集群中其他节点，若大多数节点同意该命令，则apply该命令到状态机执行，返回执行结果给client，并在下次appendentries时告诉其他节点apply到状态机执行

raft 算法中有两种timeout:

  • election_timeout: follower在该时长内没有收到leader的heartbeat消息，则成为candidate / 一轮election在election_timeout间隔内没成功,candidate重新选举
  • heartbeat_timeout: leader每隔heartbeat_timeout时间向其他节点发送appendentries request

实现分析:

  • 称每个节点为一个server，在server.py定义了一个Server类，表示节点，每个节点只有一个Server类的实例，存储节点的各种属性状态
  • 由于每个节点的状态可能会在 follower/candidate/leader 三种不同的状态间切换，不同状态又对应了不同的响应操作，因此把节点状态单独抽离了出来，分别对应 state.py 中的 Follower/Candidate/Leader 三个类。当前节点的状态通过当前Server实例的state属性定义；三种状态中通过set_server方法来关联当前节点，通过self._serverget&set当前节点的各种属性值。通过这种方式，实现了节点运行过程中的状态的动态改变。
  • 该部分主要有两个线程:
    • 主动逻辑线程：不同state下主动执行的逻辑，如follower的超时选举，candidate发起requestVote请求，leader发起appendEntries请求，及节点状态之间的改变等
    • 被动响应线程：负责消息监听，当其他节点发来的消息，执行对应逻辑，返回给发送方消息。如follower响应requestVote请求，给其他节点投票；响应appendEntries请求，重置timeout/append log entry/apply entry to state machine等
  • 这部分比较关键的一点是timeout的reset，follower等待election_timeout的过程中，收到了candidate的requestVote请求或者leader的appendEntries请求时，要reset election_timeout，而等待timeout的逻辑是在主动逻辑线程，timeout的reset是在被动响应线程，这就要涉及到进程的通信，这里我用到了 threading模块的Event，通过set()就能终止正在进行的timeout wait，执行下面的逻辑。

流程梳理：

  • 当一个server起来的时候，初始默认状态是follower，在election_timeout的时间内，有以下几种情况：
    • 没有其他节点发来的消息，则成为candidate
    • candidate发来requestVote请求，重置timeout，响应请求
    • leader发来appendentries请求，重置timeout，响应请求
  • 当server的state变为candidate，当前任期加一，给自己投一票，立即向其他节点发起requestVote请求，在timeout时间内，有以下几种情况：
    • 收到了大多节点给自己当前任期(包括自己)的投票，成为leader
    • 没有收到大多数节点(包括自己)的投票，timeout结束开始新一轮竞选
  • 当server的state变为leader，立即向其他节点发起appendentries请求，告诉集群中其他节点当前的leader是自己
  • 对于所有状态下的节点: 当发来的消息的任期大于server的当前任期，不管节点的当前状态是什么，立即变为follower，设置当前任期为消息传来的任期

Log replication

实现分析

  • log replication 实际还是通过appendentries请求实现的
  • 大体过程：当一个节点成为leader后，他会不断的向其他节点发送heartbeat(空entry的appendentries请求)。当有客户端请求leader节点时，会先将客户端命令command加到节点的log列表中，在下次appendEntries请求中将command传递给其他节点，当大多数节点响应该请求将command追加到自身的log列表中时，commit该command到state_machine执行，返回结果给客户端，并在下次appendentries请求中告诉其他节点commit该command到state_machine并执行。
  • 实现细节：
    • leader 的nextIndex{}和matchIndex{} 为其他节点分别维护了将要发送给他的下一个log entry的index(默认leader的最后log entry的index + 1)、已知的复制到他的log entry的最大index(默认0)(PS: index start from 1)
    • append_entries取出每个节点的nextIndex，并把他们的前一个entry的prevLogIndex和prevLogTerm，及nextIndex位置以后的entryies发送给他们，若节点的prevLogIndex位置有entry并且Term和prevLogTerm匹配，便在自身log列表的prevLogIndex后追加entries，回送消息告诉leader prev匹配，否则回送消息告诉leader prev不匹配。
    • 若某节点的prev log entry匹配，则更新leader的matchIndex中该节点的match index并更新nextIndex为leader的last log index + 1;
    • 若节点的prev log entry不匹配，则更新leader中nextIndex该节点的next index - 1，若后续的append_entries仍然返回不匹配的结果，重复执行该步骤更新该节点对应的nextIndex直至匹配。
